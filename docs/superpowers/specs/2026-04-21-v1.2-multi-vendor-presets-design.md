# v1.2 Design — Multi-Vendor Preset Ergonomics

**Status:** Approved (brainstorming complete 2026-04-21)
**Target release:** v1.2
**Scope:** Additive ergonomics — no breaking changes. Makes 8 OpenAI-compatible vision vendors accessible without authoring a `vision-config.json`.

---

## 1. Goals

1. **Zero-config activation for preset vendors.** A user who wants Moonshot vision sets `VISION_DEFAULT_PROVIDER=moonshot` and `MOONSHOT_API_KEY=...` — nothing else — and the server registers Moonshot as an OpenAI-compatible provider with a sensible default model and preset base URL.
2. **No new namespace, no collision risk.** Preset activation is opt-in via the existing `VISION_DEFAULT_PROVIDER` selector. Vendor env vars (`MOONSHOT_API_KEY` etc.) are read **only** when the user has explicitly named that preset. Host runtimes that happen to have `MOONSHOT_API_KEY` set for their own purposes are not silently co-opted.
3. **Power-user path preserved.** Multi-preset setups, custom models, and non-OpenAI-compatible vendors (e.g., Minimax) remain fully supported through `vision-config.json` with the existing `${ENV_VAR}` interpolation.
4. **Copy-paste docs.** Each preset gets a ready-to-paste JSON snippet in the README so users transitioning from "just works via env" to "custom per-provider settings" don't start from scratch.

## 2. Non-goals (out of scope for v1.2)

- **Auto-scanning preset env vars at startup.** Rejected during brainstorming: too aggressive, and reintroduces a narrower form of the host-collision concern the design is built to avoid.
- **Multi-preset activation via env vars only.** Users wanting both Moonshot and Zhipu registered simultaneously write a `vision-config.json`. Out of scope: `VISION_EXTRA_PROVIDERS` or similar enumeration env vars.
- **New provider class for non-OpenAI-compatible vendors.** Minimax's native shape differs from OpenAI's chat-completions format; it stays on `GenericHttpProvider` via config file. No new adapter class in v1.2.
- **Per-preset retry/concurrency tuning.** Presets inherit the existing global defaults; power users override via config file.
- **Streaming, structured output, caching, fallback chains.** Still tracked as v1.3+ candidates in `feature_ideas.md` memory; not touched here.

## 3. Behavioral contract

The shortest path from zero to a working Moonshot call:

```bash
export VISION_DEFAULT_PROVIDER=moonshot
export MOONSHOT_API_KEY=sk-...
llm-vision-mcp
# Server logs: "llm-vision-mcp: 1 provider(s) active: moonshot"
# Server logs: "llm-vision-mcp: default provider: moonshot"
```

Optional overrides:
- `MOONSHOT_MODEL=moonshot-v1-32k-vision-preview` — overrides preset's default model.
- `MOONSHOT_BASE_URL=https://api.moonshot.cn/v1` — overrides preset's default base URL (use case: Chinese region, gateway, self-hosted proxy).

Equivalent CLI shape:

```bash
llm-vision-mcp --provider moonshot --api-key $MOONSHOT_API_KEY \
  --model moonshot-v1-32k-vision-preview \
  --base-url https://api.moonshot.cn/v1
```

The generic `--api-key` and `--base-url` flags set whichever provider is currently configured as default. They work for builtins and presets alike.

**Precedence** (unchanged from today, documented here for completeness):
1. Preset defaults (baked into `src/presets.ts`).
2. Vendor env vars (`MOONSHOT_API_KEY`, `MOONSHOT_MODEL`, `MOONSHOT_BASE_URL`).
3. Generic CLI flags (`--api-key`, `--model`, `--base-url`, `--timeout`).
4. `vision-config.json` `providers.<name>` entry.

If step 4 defines `providers.moonshot`, the file entry's explicit fields win — but preset defaults still fill any gaps the file leaves unspecified. So a file entry with just `{ "apiKey": "${KEY}" }` under a preset name still gets `baseUrl` and `model` from the preset. This is a deliberate departure from strict "file wins in its entirety" because a user naming a preset in their file is signaling they want preset behavior plus their overrides.

## 5. Architecture

Three files touched; one new.

### 5.1 New: `src/presets.ts`

Pure data plus a shape contract. No control flow.

```ts
export interface Preset {
  baseUrl: string;
  defaultModel: string;
  envKey: string;
  envModel: string;
  envBaseUrl: string;
}

export const PRESETS: Record<string, Preset> = {
  moonshot: {
    baseUrl: "https://api.moonshot.ai/v1",
    defaultModel: "moonshot-v1-8k-vision-preview",
    envKey: "MOONSHOT_API_KEY",
    envModel: "MOONSHOT_MODEL",
    envBaseUrl: "MOONSHOT_BASE_URL",
  },
  // ...7 more entries
};

export function getPreset(name: string): Preset | undefined {
  return PRESETS[name];
}
```

### 5.2 Modified: `src/config.ts`

A new helper `applyPresetDefaults(config)` runs at the end of `loadConfig`, after env → CLI → file merging. It synthesizes a preset-shaped entry for `config.providers[config.defaultProvider]` when (a) the default provider name matches a preset, and (b) an API key is available from either CLI flags or the vendor env var.

Precedence within the helper: CLI-supplied values (already sitting on `existing`) > vendor env vars > preset defaults. Config file entries have already been merged before this helper runs, and since file entries are authoritative in the existing precedence chain, the helper should **skip any field the file explicitly set** — it only fills blanks.

```ts
function applyPresetDefaults(config: AppConfig): void {
  const name = config.defaultProvider;
  const preset = PRESETS[name];
  if (!preset) return;
  const existing = config.providers[name];
  // If the file entry has explicitly chosen a non-OpenAI-compatible type
  // (e.g., generic-http), respect that — the user knows what they're doing.
  if (existing?.type && existing.type !== "openai-compatible") return;
  const apiKey = existing?.apiKey ?? process.env[preset.envKey];
  if (!apiKey) return;
  config.providers[name] = {
    type: "openai-compatible",
    apiKey,
    baseUrl: existing?.baseUrl ?? process.env[preset.envBaseUrl] ?? preset.baseUrl,
    model: existing?.model ?? process.env[preset.envModel] ?? preset.defaultModel,
    ...(existing?.timeout !== undefined ? { timeout: existing.timeout } : {}),
    ...(existing?.retry !== undefined ? { retry: existing.retry } : {}),
    ...(existing?.concurrency !== undefined ? { concurrency: existing.concurrency } : {}),
  };
}
```

Edge cases the helper handles by no-op:
- Default provider name isn't a preset (e.g., `openai`, `fopenai`, or an unknown string) → returns immediately.
- Preset matched but no API key available anywhere → returns without synthesizing; the downstream "no providers configured" startup error takes over.
- File entry under a preset name has `type: "generic-http"` → helper respects the user's explicit choice and doesn't clobber.

**File-entry interaction:** If `vision-config.json` defines `providers.<preset-name>` with *all* required fields, `existing` is already complete and the helper just refills from it — effectively a no-op. If the file entry is *partial* (e.g., only `apiKey`), the helper fills the gaps with preset defaults. This is a slight departure from strict "file wins in its entirety" but it's the more forgiving behavior, and users writing a file entry for a preset name almost always want preset defaults filled in.

### 5.3 Modified: `src/config.ts` — CLI flags

Add two generic flags:

- `--api-key <key>` → sets `apiKey` on `config.providers[config.defaultProvider]`.
- `--base-url <url>` → sets `baseUrl` on `config.providers[config.defaultProvider]`.

**Ordering matters.** `config.defaultProvider` is resolved in step 3 of `loadConfig` from `--provider`. The generic `--api-key` / `--base-url` flags must be applied after that step, so that `--provider moonshot --api-key X` writes the key into `providers.moonshot` (not into whatever `VISION_DEFAULT_PROVIDER` pointed at before CLI parsing). This means these flags are applied just before `applyPresetDefaults` runs, and after vendor-specific flags like `--openai-api-key` have already seeded their own entries.

These are intentionally generic — they don't duplicate the vendor-specific `--openai-api-key`, `--anthropic-api-key`, `--google-api-key` flags, which remain for backward compatibility. The generic flags scale to N presets without per-vendor clutter.

### 5.4 Unchanged: `src/index.ts`

No changes. `OpenAICompatibleProvider` already consumes the shape we synthesize (`type: "openai-compatible"` + `baseUrl` + `apiKey` + `model`).

## 6. Error handling

### 6.1 Unknown provider name

`VISION_DEFAULT_PROVIDER=fopenai` (typo) with no config-file entry:

```
Error: Unknown provider "fopenai".
  Known presets: moonshot, zhipu, qwen, nvidia, groq, together, deepinfra, xai
  Known builtins: openai, anthropic, google, ollama
  Or provide a vision-config.json entry for "fopenai".
```

Handled by an early check in `src/index.ts`'s `main()` after `buildRegistry` sees zero providers. The error message enumerates both sets to make the fix obvious.

### 6.2 Preset named but API key unset

`VISION_DEFAULT_PROVIDER=moonshot` with no `MOONSHOT_API_KEY`:

The preset resolution helper no-ops (no key, nothing to synthesize). `buildRegistry` sees zero providers and exits with the existing error, augmented to point at the preset:

```
Error: No providers configured. Preset "moonshot" requires MOONSHOT_API_KEY to be set.
  Or pass --api-key on the command line.
  Or define providers.moonshot in vision-config.json.
```

### 6.3 Preset + file entry with same name

Config file `providers.moonshot` exists. `applyPresetDefaults` runs after the file merge and sees `existing` populated from the file. Any fields the file explicitly set (e.g., `apiKey`, `model`) are preserved; fields the file omitted (e.g., `baseUrl`) are filled in from the preset. Silently — no warning log. (Future consideration: emit an info-level log when a preset name is partially overridden by file config, but not v1.2 scope.)

### 6.4 Default provider name doesn't resolve to any registered provider

Edge case: user has `OPENAI_API_KEY` set (so openai registers successfully) but also sets `VISION_DEFAULT_PROVIDER=fopenai` (typo — not a preset, not a builtin, no file entry). Today this silently passes startup because `buildRegistry` produces one provider and `main()` only errors on zero providers. The first MCP request without an explicit provider parameter then fails confusingly with "Unknown provider: fopenai."

Add a startup validation in `main()` (src/index.ts): after `buildRegistry`, check that `registry.listProviders().includes(config.defaultProvider)`. If not, fail startup with the §6.1 error message. This covers both the "typo" case and the "preset named but key unset" case uniformly.

## 8. Testing

### 8.1 `tests/presets.test.ts` (new)

Structural tests on the preset table itself — these catch typos when adding new presets:

- Every preset has `baseUrl`, `defaultModel`, `envKey`, `envModel`, `envBaseUrl` (non-empty strings).
- Every `baseUrl` parses via `new URL(...)` without throwing.
- No two presets share the same `envKey` (prevents ambiguous env reads).
- `envModel` and `envBaseUrl` both start with the same prefix as `envKey` up to the final `_*` segment (catches copy-paste drift).

### 8.2 `tests/config.test.ts` (extended)

New test cases under a `describe("preset resolution")` block:

- `VISION_DEFAULT_PROVIDER=moonshot` + `MOONSHOT_API_KEY=test-key` → `config.providers.moonshot` exists with type, apiKey, baseUrl=preset default, model=preset default.
- Env `MOONSHOT_MODEL` overrides preset default model.
- Env `MOONSHOT_BASE_URL` overrides preset default baseUrl.
- CLI `--provider moonshot --api-key X` synthesizes a full preset entry.
- CLI `--model Y` on top of preset wins over `MOONSHOT_MODEL` env.
- CLI `--base-url Z` on top of preset wins over `MOONSHOT_BASE_URL` env.
- Config file entry `providers.moonshot` with partial fields gets preset defaults merged in for missing fields (`baseUrl`, `model`) but keeps the file's explicit values (`apiKey`, etc.).
- Config file entry `providers.moonshot` with all fields fully specified keeps all of them — preset defaults are not re-applied over explicit values.
- Unknown `VISION_DEFAULT_PROVIDER` value produces no synthesized entry (the downstream `main()` error handles reporting).
- Preset named but no API key anywhere → no synthesized entry.
- `zhipu` preset uses `ZHIPUAI_API_KEY` (not `ZHIPU_API_KEY`) — regression guard on the non-uniform naming.
- `qwen` preset uses `DASHSCOPE_API_KEY` — same.

### 8.3 `tests/index.test.ts` (new, small)

Two assertions, sealing the config-to-registry wiring:

- When `config.providers.moonshot` is synthesized with `type: "openai-compatible"`, `buildRegistry` constructs an `OpenAICompatibleProvider` with the expected `baseURL`.
- When `config.defaultProvider` names a provider that isn't in `registry.listProviders()`, startup fails with the §6.1 error message (covers the §6.4 regression for both typos and missing API keys).

Not included: live HTTP tests. `openai-compatible.test.ts` already exercises the OpenAI-compatible transport.

## 9. Migration and backward compatibility

- **No breaking changes.** All new surface is additive.
- **Existing `--openai-api-key`, `--anthropic-api-key`, `--google-api-key`, `--ollama-*` flags continue to work.** The new generic `--api-key` and `--base-url` flags are alternatives, not replacements.
- **Users with existing `vision-config.json` files are unaffected** — their file entries continue to take precedence.
- **Preset names are chosen to not conflict with builtin names** (`openai`, `anthropic`, `google`, `ollama`). If a future vendor's natural name collides, we'll document that edge.

## 11. Open questions (tracked, not blocking)
