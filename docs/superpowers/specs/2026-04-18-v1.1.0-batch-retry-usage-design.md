# v1.1.0 Design — Batch, Retry, Usage Reporting

**Status:** Approved (brainstorming complete 2026-04-18)
**Target release:** v1.1.0
**Scope:** Three additive features to `llm-vision-mcp` — no breaking changes at the MCP tool surface.

---

## 1. Goals

1. **Batch processing** — single tool call describes N images in parallel with per-item prompt/provider/model override.
2. **Retry with backoff** — all provider calls transparently retried on transient errors (network, 429, 5xx).
3. **Token/cost reporting** — SDK-exposed usage data surfaced alongside every description.

## 2. Non-goals (out of scope for v1.1.0)

- New tools beyond `describe_images` (no `extract_text`, `analyze_chart`, `compare_images`).
- Structured output / JSON schemas.
- Response caching.
- Provider fallback chains.
- Streaming responses.
- Named prompt templates.
- CLI/env-var exposure of retry and concurrency settings (config-file only).

## 3. Provider interface change

Single invasive change: widen the `VisionProvider.describeImage` return type to carry usage data.

**Before:**
```ts
interface VisionProvider {
  describeImage(input: ImageInput, options: DescribeOptions): Promise<string>;
}
```

**After:**
```ts
interface VisionProvider {
  describeImage(input: ImageInput, options: DescribeOptions): Promise<VisionResult>;
}

interface VisionResult {
  text: string;
  usage?: UsageInfo;
}

interface UsageInfo {
  inputTokens: number;
  outputTokens: number;
  totalTokens: number;
}
```

Per-provider mapping:

| Provider | Source field |
|---|---|
| Anthropic | `response.usage.input_tokens` / `output_tokens` |
| OpenAI | `response.usage.prompt_tokens` / `completion_tokens` / `total_tokens` |
| OpenAI-compatible | Same as OpenAI (SDK shape) |
| Google | `result.response.usageMetadata.promptTokenCount` / `candidatesTokenCount` / `totalTokenCount` |
| Ollama | `response.prompt_eval_count` / `eval_count`; `usage: undefined` if either missing |
| Generic HTTP | Optional `usagePath` config field — extract `totalTokens` from response JSON if configured; else `usage: undefined` |

When mapped fields are all present, `totalTokens` is the provider-reported total when available, or `inputTokens + outputTokens` otherwise.

## 4. Retry wrapper

New file `src/retry.ts`. Wraps `provider.describeImage` at the registry boundary so all six providers share identical semantics without per-provider code.

```ts
interface RetryConfig {
  maxAttempts: number;
  baseDelayMs: number;
  maxDelayMs: number;
  jitterFactor: number;
}

async function withRetry<T>(
  fn: () => Promise<T>,
  config: RetryConfig,
  isRetryable: (err: unknown) => boolean,
): Promise<T>;
```

**Defaults:**
```ts
const DEFAULT_RETRY: RetryConfig = {
  maxAttempts: 3,
  baseDelayMs: 500,
  maxDelayMs: 10_000,
  jitterFactor: 0.2,
};
```

**Retryable predicate** returns `true` for:
- Network errors (fetch rejection, abort, DNS failure, ECONNRESET, socket hangup)
- HTTP 429 responses
- HTTP 5xx responses
- SDK errors carrying `.status` matching the above (Anthropic SDK `APIError`, OpenAI SDK `APIError`)

**Not retryable** (error propagates immediately):
- 400 invalid request
- 401 / 403 auth
- 404 not found
- Content-policy rejections (provider-specific detection best-effort)
- Errors without `.status` that aren't network errors (preprocessing errors, config errors)

**Backoff:** `delay = min(baseDelayMs * 2^(attempt-1), maxDelayMs) * (1 ± jitterFactor)`.
**Rate-limit awareness:** if the retryable error exposes a `Retry-After` header, use that value (clamped to `maxDelayMs`) instead of the computed backoff for the next attempt. Best-effort only — depends on the SDK surfacing headers. OpenAI SDK `APIError`, Anthropic SDK `APIError`, and our hand-rolled fetch providers (Ollama, generic HTTP) expose headers; Google SDK does not, so Google 429s fall back to computed backoff.

**Coverage caveat:** the retryable predicate requires `.status` or network-error identification. Google SDK errors that wrap HTTP responses without a `.status` field will not be retried. This is acceptable for v1.1.0 — if it becomes a pain point, v1.2+ can add a per-SDK adapter.

**Integration:** `ProviderRegistry.getProvider(name)` returns a `RetryingProvider` wrapper. The wrapper holds the resolved retry config (provider-level override layered over global) and delegates all other fields unchanged. Providers themselves are untouched.

## 5. Batch tool

New MCP tool `describe_images` alongside existing `describe_image`. Existing tool behavior unchanged.

### Input schema (Zod)

```ts
{
  items: z.array(z.object({
    image: z.string(),
    prompt: z.string().optional(),
    provider: z.string().optional(),
    model: z.string().optional(),
  })).min(1).max(100),
  prompt: z.string().optional(),
  provider: z.string().optional(),
  model: z.string().optional(),
  concurrency: z.number().int().positive().optional(),
}
```

### Resolution chains

For each item, resolve values in this order (first non-null wins):
- **prompt:** `item.prompt → args.prompt → provider.defaultPrompt`
- **provider:** `item.provider → args.provider → config.defaultProvider`
- **model:** `item.model → args.model → provider default`

### Concurrency

`concurrency` resolution order: `args.concurrency → provider.concurrency config → 3`.

One semaphore **per resolved provider name** — items destined for different providers run in parallel pools that don't block each other (e.g., OpenAI at 3 concurrency doesn't starve Ollama at 1).

The concurrency slot is held for the entire lifetime of an item, including retry delays. A retrying item does not release and re-acquire.

### Execution — `src/batch.ts`

```ts
async function describeImagesBatch(
  args: BatchArgs,
  registry: ProviderRegistry,
  preprocessingOptions: PreprocessingOptions,
): Promise<BatchResult>;
```

Per item:
1. Acquire provider-specific semaphore slot.
2. Resolve provider, prompt, model.
3. `resolveImage(item.image)` then `preprocessImage(...)` against the resolved provider's `supportedFormats`.
4. Call retry-wrapped `provider.describeImage(...)`.
5. Record `{ index, description, usage }` on success or `{ index, image, error }` on failure.
6. Release slot.

Items execute with `Promise.all` over the item list; per-item errors are caught and converted to error result objects (they do not reject the outer promise).

### Output

Single MCP text content block containing JSON:

```json
{
  "summary": { "succeeded": 9, "failed": 1 },
  "totalUsage": { "inputTokens": 12340, "outputTokens": 580, "totalTokens": 12920 },
  "results": [
    { "index": 0, "description": "A red square", "usage": { "inputTokens": 1250, "outputTokens": 42, "totalTokens": 1292 } },
    { "index": 1, "description": "A blue circle", "usage": { "inputTokens": 1180, "outputTokens": 38, "totalTokens": 1218 } },
    { "index": 2, "image": "chart.png", "error": "Timeout after 3 retries" }
  ]
}
```

- `summary` front-loaded so the consuming LLM notices failures without scanning all items.
- `totalUsage` sums only successful items. A successful item without `usage` contributes zero to the sum. If no successful item reports usage, the `totalUsage` field is omitted from the output entirely.
- `index` is preserved because items can complete out of order due to concurrency.
- The tool itself returns `isError: undefined` (not failed) even when all items fail — the JSON body communicates the failures. Only Zod schema violations at the MCP input layer cause `isError: true`.

## 6. Usage at the single-image tool

The existing `describe_image` tool also surfaces usage. MCP tool result gains a second content block when usage is present:

```json
{
  "content": [
    { "type": "text", "text": "A red square on white background" },
    { "type": "text", "text": "Usage: 1250 in / 42 out / 1292 total tokens" }
  ]
}
```

When `usage` is `undefined` (Ollama edge cases, generic HTTP without `usagePath`), only the description block is returned — existing single-block consumers still work.

## 7. Config schema additions

All additions are optional — existing `config.json` files continue to load unchanged.

```ts
interface AppConfig {
  defaultProvider: string;
  providers: Record<string, ProviderConfig | GenericHttpProviderConfig>;
  preprocessing: PreprocessingOptions;
  retry?: Partial<RetryConfig>;          // NEW — global defaults
}

interface ProviderConfig {
  // ...existing...
  retry?: Partial<RetryConfig>;          // NEW — per-provider override
  concurrency?: number;                  // NEW — per-provider batch concurrency cap
}

interface GenericHttpProviderConfig extends ProviderConfig {
  // ...existing...
  usagePath?: string;                    // NEW — optional JSON path for usage total tokens
}
```

`DEFAULT_RETRY` and `DEFAULT_CONCURRENCY = 3` exported from `types.ts`.

Example `config.example.json` additions:
```json
{
  "retry": { "maxAttempts": 3, "baseDelayMs": 500 },
  "providers": {
    "ollama": {
      "type": "openai-compatible",
      "baseUrl": "http://localhost:11434/v1",
      "retry": { "maxAttempts": 1 },
      "concurrency": 1
    }
  }
}
```

## 8. Testing

TDD for each feature.

**Provider interface widening** — per provider (6 × ~2 tests):
- Returns `usage` mapped from SDK response.
- Returns `usage: undefined` when the SDK response lacks usage fields.
- Existing assertions updated to check `result.text` instead of the bare string.

**Retry wrapper** — `tests/retry.test.ts`:
- First attempt success: `fn` called once, zero delay.
- Retries on 429, honors `Retry-After` (clamped to `maxDelayMs`).
- Retries on 5xx.
- Does NOT retry on 400 / 401 / 403 / 404.
- Does NOT retry on preprocessing errors (no `.status`, not a network error).
- Exhaustion: throws the last error after `maxAttempts`.
- Jitter bounds: 10-run statistical check within expected range.
- Uses fake timers (`vi.useFakeTimers()`) for deterministic backoff timing.

**Batch** — `tests/batch.test.ts`:
- Uniform prompt, 3 items, all succeed.
- Per-item prompt override honored.
- Per-item provider override splits items across two mocked providers.
- Partial failure: middle item throws, result contains 2 successes + 1 error object.
- Concurrency limit enforced (mock with an externally-controlled gate, assert max in-flight).
- Per-provider concurrency isolation: OpenAI items (concurrency=3) proceed while Ollama (concurrency=1) serializes.
- `totalUsage` sums successful items only.
- `index` preserved across out-of-order completion.

**Server integration** — `tests/server.test.ts` additions:
- `describe_images` tool registered with correct schema.
- Input validation: rejects empty `items`, rejects `items.length > 100`.
- Existing `describe_image` tool still works, now emits usage content block when usage is present.

Target: preserve current test coverage level (no regressions).

## 10. Migration / compatibility

- Users upgrading from 1.0.x see no change to `describe_image` output unless their provider now reports usage (previously silently dropped). The usage content block is additive.
- Config files without `retry` or `concurrency` pick up new defaults automatically (3 attempts, concurrency 3).
- `VisionProvider` interface change is internal — the project doesn't expose it as a public API.
- Semver: **minor bump to v1.1.0**. No breaking changes at the MCP tool level.
